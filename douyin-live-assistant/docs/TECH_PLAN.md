# 抖音直播挂机助手 - 技术方案文档

## 项目概述

| 项目 | 内容 |
|------|------|
| 项目名称 | 抖音直播挂机助手 |
| 目标 | 自动化登录抖音账号，在指定直播间挂机捧场（增加人气、点赞） |
| 阶段 | 初期支持5账号同时在线 → 后期扩展至1000+账号矩阵 |

---

## 一、需求分析

### 1.1 核心功能

| 优先级 | 功能 | 描述 |
|--------|------|------|
| P0 | 账号授权 | 扫码登录，保存cookies，避免重复登录 |
| P0 | 直播间挂机 | 自动进入指定直播间，保持在线状态 |
| P0 | 自动点赞 | 随机间隔点赞，模拟真人行为 |
| P1 | 账号管理 | 多账号切换、状态监控 |
| P1 | 防检测机制 | 随机延迟、行为模拟、设备指纹混淆 |
| P2 | IP代理池 | 每个账号配置独立代理IP（防止IP封禁） |
| P2 | 批量矩阵 | 1000+账号同时运行 |

### 1.2 用户交互流程

```
1. 用户扫码授权 → 2. 配置目标直播间 → 3. 启动挂机任务 → 4. 监控运行状态
```

### 2.2 技术栈

| 层级 | 技术选型 | 理由 |
|------|----------|------|
| 语言 | Python 3.10+ | 生态丰富，易于维护 |
| 浏览器自动化 | Playwright | 比Selenium更稳定，反检测能力更强 |

| 数据库 | SQLite | 轻量，无需额外服务 |
| 代理管理 | ProxyPool / 付费代理API | 住宅代理/手机流量卡 |

## 三、核心模块设计

### 3.1 账号授权模块

**功能**：扫码登录，保存cookies

**实现方式**：
1. 提供Web页面显示登录二维码
2. 用户扫码后，Playwright自动捕获cookies
3. 存储cookies到本地文件（加密）

**文件结构**：
```
cookies/
├── account_001.json    # 账号1的cookies
├── account_002.json    # 账号2的cookies
└── ...
```

### 3.2 挂机执行模块

**核心逻辑**：
```python
class LiveRoomRunner:
    def run(self, account_id, room_id):
        # 1. 加载cookies，登录抖音
        # 2. 进入直播间 (https://live.douyin.com/{room_id})
        # 3. 循环执行：
        #    - 随机延迟 (3-10秒)
        #    - 点击点赞按钮
        #    - 偶尔滑动页面 (模拟浏览)
        # 4. 每隔30分钟刷新页面 (保持在线)
```

### 3.3 防检测机制

| 措施 | 实现方式 |
|------|----------|
| 随机延迟 | 点赞间隔 3-10秒 随机 |
| 行为随机化 | 随机滑动、随机滚动 |
| 浏览器指纹 | 随机UA、分辨率、时区、语言 |
| 运行时长 | 单账号每次不超过 3 小时 |
| 账号轮换 | 定期更换登录账号 |

### 3.4 代理IP管理（后期扩展）

**架构**：
```
代理池 ←→ 账号绑定 ←→ 执行器
```

**选型建议**：
- 住宅代理（稳定但贵）
- 手机流量卡（便宜，需自己维护）
- 付费代理API（如 luminati、brightdata）

## 八、风险与注意事项

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 账号被封 | 高 | 降低点赞频率，控制运行时长，使用代理IP |
| IP被封 | 中 | 使用住宅代理，多IP轮换 |
| 行为被检测 | 中 | 增强随机化，模拟真人操作 |

### 重要提醒

1. **不要高频点赞**：每分钟不超过10次
2. **控制运行时长**：单账号每天不超过6小时
3. **使用代理IP**：每个账号建议独立IP
4. **账号质量**：新账号容易被封，建议用老号
